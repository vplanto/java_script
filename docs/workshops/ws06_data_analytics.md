# Воркшоп 6: Від гри до Data-Driven платформи (Збереження та Аналітика)

Ваша гра працює. AI підлаштовується під гравця, хешування захищає паролі, а інтерфейс виглядає сучасно. Але є одна критична архітектурна проблема: **ваша гра страждає на амнезію**.

Щоразу, коли користувач оновлює сторінку (F5), усі епічні битви, рекорди та навіть стан авторизації зникають, немов їх ніколи й не було. 

У цьому фінальному модулі ми вирішимо цю проблему і перетворимо нашу гру на справжню **аналітичну платформу**. Ми навчимо її запам'ятовувати, збирати телеметрію та експортувати Big Data для наших "бізнес-партнерів".

---

## Акт 1: Проблема Амнезії (Persistence)

Нам потрібна база даних. Але ми пишемо клієнтський (Front-end) додаток, у нас немає сервера. Що робити? Використовувати кишеню браузера — **LocalStorage**.

Щоб гра "пам'ятала" користувачів та їх історію, ми маємо серіалізувати наші об'єкти в рядки і записати їх у браузер.

**Ваше завдання:**
Додайте функції збереження та завантаження стану гри:

```javascript
// Збереження та Завантаження
function saveToStorage() {
    if (storageAvailable('localStorage')) {
        localStorage.setItem('users', JSON.stringify(users));
        localStorage.setItem('gameHistory', JSON.stringify(gameHistory));
    }
}

function loadFromStorage() {
    if (storageAvailable('localStorage')) {
        const storedUsers = localStorage.getItem('users');
        const storedHistory = localStorage.getItem('gameHistory');
        
        if (storedUsers) users = JSON.parse(storedUsers);
        if (storedHistory) gameHistory = JSON.parse(storedHistory);
    }
}
```

Тепер після перезавантаження сторінки гравець залишається залогіненим, а історія ігор відновлюється!

## Акт 2: Народження Аналітики (Телеметрія)

Якщо ми вже вміємо зберігати дані, чому б не зберігати **все**? 
У реальних проектах (як-от Netflix чи Spotify) розробники логують кожну дію користувача. Це називається **Телеметрія**. 

Замість того, щоб просто збільшувати лічильник перемог, ми створимо детальний масив логів `gameHistory`. 

**Ваше завдання:**
Після кожного раунду додавайте до масиву `gameHistory` новий об'єкт-подію. 

```javascript
const logEntry = {
    timestamp: new Date().toISOString(), // Точний час події у міжнародному стандарті
    user: currentUser.nickname,          // Хто грав
    aiLevel: "adaptive",                 // Режим гри
    playerMove: "Rock",                  // Що обрав гравець
    aiMove: "Paper",                     // Що відповів AI
    result: "loss"                       // Фінальний результат
};
gameHistory.push(logEntry);
saveToStorage(); // Одразу зберігаємо
```

> [!TIP]
> **Навіщо тут ISO Date (`toISOString`)?**
> Щоб дані були стандартизовані. Якщо ваш гравець у Японії, а сервери в США, ви не хочете розбиратися з часовими поясами. ISO 8601 (`2026-04-20T14:30:00.000Z`) — це інженерний стандарт, зрозумілий будь-якій системі.

## Акт 3: Корпоративний Замовник (JSON vs XML)

Уявіть, що ваша гра стала хітом. Вами зацікавились інвестори. 

1. **Сучасний стартап-інвестор** просить ваші аналітичні дані у форматі **JSON**.
2. **Старий консервативний банк** також хоче купити ваші дані, але їхні архаїчні системи розуміють **ТІЛЬКИ XML**.

### JSON Export

Це просто. Ви створюєте `Blob` (Binary Large Object) з типом `application/json` і генеруєте посилання на скачування .

### XML Export (Челендж)

JS не має простого `stringify` для XML. Вам треба написати рекурсивну функцію-конвертер.

**Алгоритм (спрощено):**

1. Якщо значення — **об'єкт** (не масив): відкрити кореневий тег, для кожного поля рекурсивно додати дочірні вузли, закрити тег.
2. Якщо значення — **масив**: для кожного елемента повторити тег із іменем поля (наприклад, кілька `<move>...</move>`).
3. Якщо значення — **примітив** (`string`, `number`, `boolean`): вміст тега екранувати для XML (`&`, `<`, `>` тощо).

### Базовий код конвертера

Використайте цей код як базу:

### Легкий шлях (JSON)
З JSON все просто. JavaScript нативно вміє перетворювати об'єкти на JSON.
```javascript
const jsonData = JSON.stringify(gameHistory, null, 2); // 'null, 2' робить красиві відступи
```

### Шлях страждань (XML Челендж)
В JS **немає** вбудованої функції `JSON.toXML()`. Чому? Бо XML складніший: там є теги, атрибути, вкладеності. Вам доведеться написати рекурсивний конвертер власноруч!

**Алгоритм конвертера:**
1. Отримуємо ключ і значення.
2. Якщо значення — об'єкт, викликаємо функцію знову (рекурсія).
3. Якщо значення — масив, проходимось по кожному елементу.
4. Якщо значення — примітив (число, рядок), загортаємо його в тег `<key>значення</key>`.

**Базова реалізація (додайте у свій код):**
```javascript
function jsonToXml(data, root = 'root', itemTag = 'item') {
    let xml = `<${root}>\n`;
    
    for (const key in data) {
        const value = data[key];
        // Якщо це масив, обробляємо кожен елемент окремим тегом itemTag
        if (Array.isArray(value)) {
            value.forEach(item => {
                xml += jsonToXml(item, itemTag, 'subitem');
            });
        } 
        // Якщо це об'єкт (не null і не масив) - рекурсія
        else if (typeof value === 'object' && value !== null) {
            xml += jsonToXml(value, key, 'subitem');
        } 
        // Якщо примітив
        else {
            // Базове екранування (escape) для XML
            const safeValue = String(value)
                .replace(/&/g, '&amp;')
                .replace(/</g, '&lt;')
                .replace(/>/g, '&gt;');
            xml += `  <${key}>${safeValue}</${key}>\n`;
        }
    }
    
    xml += `</${root}>\n`;
    return xml;
}

// Приклад:
// const xml = jsonToXml({ user: { name: "O'Brien", score: 10 } }, "game");
```

## Акт 4: Магія Завантаження (Blob)

Дані серіалізовано. Рядки JSON та XML лежать в оперативній пам'яті комп'ютера. 
Але замовник хоче натиснути кнопку **"Завантажити звіт"** і отримати реальний файл собі на диск.

Як змусити браузер "завантажити" файл, якого не існує на жодному сервері? Тут починається справжня магія Front-end'у.

**Знайомтесь: Blob (Binary Large Object)**
Ми беремо наш текст і кажемо браузеру: "Уяви, що це справжній файл". Потім ми створюємо невидиме посилання (тег `<a>`), вішаємо на нього цей фейковий файл і програмно "клікаємо" по ньому.

```javascript
function downloadData(content, fileName, contentType) {
    // 1. Створюємо "фейковий" файл у пам'яті
    const file = new Blob([content], { type: contentType });
    
    // 2. Створюємо невидиме посилання
    const a = document.createElement("a");
    
    // 3. Генеруємо тимчасову URL-адресу для нашого Blob-файлу
    a.href = URL.createObjectURL(file);
    a.download = fileName; // Вказуємо ім'я файлу при завантаженні
    
    // 4. Програмно симулюємо клік користувача
    a.click();
    
    // 5. Прибираємо за собою (очищаємо пам'ять)
    URL.revokeObjectURL(a.href);
}
```

Тепер, коли користувач натискає кнопку в інтерфейсі, викликайте:
`downloadData(jsonData, 'analytics.json', 'application/json');`

## Контрольні питання

1. Навіщо логувати події з `new Date().toISOString()`? Що дає ISO-формат дати інженеру?

<details markdown="1">
<summary>Відповідь</summary>

ISO 8601 — це універсальний стандарт без прив'язки до часового поясу. Він дозволяє легко сортувати події хронологічно та відтворювати історію дій без багів, спричинених локальним часом на комп'ютері клієнта.

</details>

2. Чому для XML не існує вбудованого аналога `JSON.stringify` у JavaScript?

<details markdown="1">
<summary>Відповідь</summary>

Тому що XML має складнішу і менш однозначну семантику (теги, атрибути, namespaces). Немає єдиного "правильного" способу перетворити масив JS в XML (робити це списками однакових тегів чи атрибутами?). Тому інженер має сам визначати правила конвертації.

</details>

3. Що таке `Blob` у контексті JavaScript і навіщо він потрібен для завантаження файлу?

<details markdown="1">
<summary>Відповідь</summary>

Blob (Binary Large Object) дозволяє створити об'єкт з бінарними або текстовими даними прямо в оперативній пам'яті браузера. За допомогою `URL.createObjectURL(blob)` ми створюємо "віртуальне" посилання на ці дані, що змушує браузер ініціювати процес скачування так, ніби це був справжній файл на сервері.

</details>
