# Воркшоп 8: Безпечна автентифікація (Auth & Security)

Ми починаємо будувати фінальний проєкт курсу — гру **"Камінь-Ножиці-Папір MMO"**.
Але будь-яка серйозна система починається не з гри, а з **Користувача**.

Ваше завдання: реалізувати систему входу, яка не зберігає паролі у відкритому вигляді.

> **Педагогіка (чому ми взагалі це робимо на клієнті):** цей воркшоп **не** вчить «як безпечно зберігати паролі у продакшені». Він показує **криптографічний примітив** (хеш) і чому **хеш на клієнті стає еквівалентом паролю** (pass-the-hash). У реальній системі пароль перевіряє **сервер**, а клієнт лише надсилає секрет по захищеному каналу — інакше будь-який XSS бачить те саме, що й «захешований» секрет.

## 1. Задача Інженера

Ми маємо форму реєстрації (Логін, Пароль).
Ми хочемо зберігати користувачів у `localStorage` (бо у нас немає бекенду).

**Проблема:** Якщо ми збережемо `{"password": "123"}`, будь-який XSS-скрипт або людина, що сіла за ваш комп'ютер, вкраде пароль.

**Рішення:** Ми будемо зберігати **Хеш** (Hash).
`"123"` -> `SHA-256` -> `"a665a45920422f9d417e4867efdc4fb8a04a1f3fff1fa07e998e86f7f7a27ae3"`

## 2. Web Crypto API (⚠️ EDUCATIONAL ONLY)

> **🚨 CRITICAL SECURITY WARNING: NEVER DO THIS IN PRODUCTION**
>
> Ми використовуємо клієнтське хешування **ТІЛЬКИ** тому, що у нас немає серверу (це навчальний проєкт із `localStorage`).
>
> **Проблема: Pass-the-Hash Attack**
> - Якщо зловмисник перехопить хеш (через Network або XSS), він може авторизуватися **БЕЗ ЗНАННЯ ПАРОЛЮ**, просто підставивши хеш.
> - Хеш, згенерований на клієнті, стає **еквівалентом паролю**.
>
> **Правильний підхід (Production):**
> 1. Клієнт відправляє **чистий пароль** по HTTPS (TLS шифрує канал).
> 2. **Сервер** хешує пароль із **Salt** (випадковий рядок, унікальний для кожного юзера) та зберігає `hash(password + salt)`.
> 3. Повторна атака неможлива: навіть якщо зловмисник викраде хеш із бази, він не зможе його використати (хеш !== пароль, а salt робить brute-force надто дорогим).
>
> **Чому сіль не «лікує» саме наш клієнтський сценарій:** salt потрібен при перевірці пароля; якщо перевірка в браузері, **і сіль, і `hash(password + salt)`** опиняються в `localStorage` (або так само доступні тому ж JS). XSS забере їх **разом** — зловмисник знову підставляє готовий секрет входу (**pass-the-hash**). На сервері salt тримається поруч із хешем **поза** досяжністю клієнтського скрипта; локально ж сіль не приховує те, що для входу ви все одно зберігаєте **вже обчислений еквівалент паролю**.
>
> Ми вчимо клієнтське хешування, щоб ви **розуміли криптографічні примітиви** (SHA-256), але **ніколи не робили це у реальних системах**.

Браузер має вбудований криптографічний модуль `window.crypto`. Використаємо його для навчальних цілей:

```javascript
async function hashPassword(password) {
    const encoder = new TextEncoder();
    const data = encoder.encode(password);
    
    // Використовуємо SHA-256 (Secure Hash Algorithm)
    const hashBuffer = await crypto.subtle.digest('SHA-256', data);
    
    // Конвертуємо бінарний буфер у шістнадцятковий рядок (hex string)
    const hashArray = Array.from(new Uint8Array(hashBuffer));
    const hashHex = hashArray.map(b => b.toString(16).padStart(2, '0')).join('');
    
    return hashHex;
}
```

> **Технічна примітка:** Цей API працює тільки в **Secure Context** (HTTPS або `localhost`). Використовуйте локальний сервер (наприклад, Live Server у VS Code).

**Інженерний аналіз:**

✅ **Що ми захистили:**
- Витік `localStorage` не видає оригінальний пароль.
- XSS-скрипт, що читає localStorage, бачить тільки хеш.

❌ **Що НЕ захищено (чому це не Production):**
- **Pass-the-Hash:** Зловмисник може скопіювати хеш і підставити його замість пароля (хеш = новий пароль).
- **Відсутність Salt:** Два юзери з однаковим паролем мають однакові хеші (Rainbow Table Attack).
- **Клієнтський код легко модифікувати:** Зловмисник може змінити JS, щоб відправляти паролі на свій сервер.

**Висновок:** Ви дізналися, як працює SHA-256 і чому **хешування — це завдання сервера, а не клієнта**. У реальних системах використовуйте bcrypt/argon2 на бекенді з унікальним salt для кожного юзера.

## Контрольні питання

1. Чому клієнтське хешування паролів (як у цьому воркшопі) небезпечне для production?

<details markdown="1">
<summary>Відповідь</summary>

Pass-the-Hash Attack: якщо зловмисник перехопить хеш (через XSS або Network), він може авторизуватися, підставивши хеш замість пароля — сервер не зможе відрізнити. Хеш стає паролем. Правильно: клієнт надсилає чистий пароль по HTTPS, сервер хешує з унікальним salt.

</details>

2. Що таке «Rainbow Table Attack» і як сіль (salt) від нього захищає?

<details markdown="1">
<summary>Відповідь</summary>

Rainbow Table — заздалегідь обчислена таблиця хешів для мільярдів популярних паролів. Якщо пароль `123456` дає хеш `X`, зловмисник миттєво знаходить пароль у таблиці.

Сіль — унікальний випадковий рядок для кожного юзера. Хешується `password + salt`. Два юзери з `123456` матимуть різні хеші. Rainbow Table марна, бо не знає salt.

</details>

3. Чому `crypto.subtle` API недоступне при відкритті файлу як `file://`?

<details markdown="1">
<summary>Відповідь</summary>

Потужні браузерні API (криптографія, камера, геолокація) доступні тільки у Secure Context — HTTPS або `localhost`. `file://` не є захищеним контекстом, бо браузер не може перевірити походження файлу. Рішення: Live Server або будь-який локальний HTTP-сервер.

</details>

4. Як перевірити пароль при логіні, якщо ми зберігаємо тільки хеш?

<details markdown="1">
<summary>Відповідь</summary>

Хешуємо введений пароль тим самим алгоритмом і порівнюємо хеші: `storedHash === await hashPassword(inputPassword)`. Хешування детерміноване — той самий вхід завжди дає той самий вихід. Порівнюємо хеші, не паролі.

</details>
